# Публікація плагінів

Усе потрібне, щоб перетворити вихідний код на підписаний бандл для розповсюдження, який користувачі зможуть сайдлоадити в Astra.

## Бандл `.astraplugin`

`.astraplugin` — це **ZIP-архів** з певною структурою. Демон валідує, (опціонально) перевіряє підпис, розпаковує його у свій каталог плагінів та запускає `entry.command` з обліковими даними.

```
<plugin-id>-<version>.astraplugin
├── plugin.toml                # Manifest
├── bin/                       # Compiled binary (Rust only)
│   └── my_plugin.exe
├── dist/                      # Bundled JS (TypeScript only)
│   └── index.js
├── src/                       # Python sources (Python only)
│   ├── plugin.py
│   └── __init__.py
├── requirements.txt           # Python deps (Python only)
├── requirements.lock          # Python deps resolved by uv (Python only)
├── ui/                        # Custom UI files (optional)
├── locales/                   # i18n JSON files (optional)
├── icon.png | icon.svg        # Optional branding
├── README.md                  # Optional
├── LICENSE                    # Optional
├── SIGNATURE                  # Ed25519 signature (if bundle is signed)
└── PUBKEY                     # Ed25519 public key (if bundle is signed)
```

Коли `astra-plugin build` створює архів, він:

1. Виконує крок збирання для відповідної мови.
2. Копіює скомпільований артефакт у очікуваний каталог.
3. Переписує `entry.command`, щоб він вказував на шлях у бандлі (лише для Rust — шляхи Python/TS всередині архіву стабільні).
4. Додає `ui/`, `locales/`, іконку та документацію, якщо вони існують поряд із `plugin.toml`.
5. Якщо існує `~/.astra/plugin-keys/private.key`, підписує кожен запис Ed25519 і додає `SIGNATURE` та `PUBKEY` до архіву.

## Підпис

### Згенерувати пару ключів

```bash
astra-plugin keygen
```

Результат:

- `~/.astra/plugin-keys/private.key` — base64-кодований seed Ed25519. **Зберігайте в таємниці.** Будь-хто з цим файлом може підписувати нові версії вашого плагіна, і користувачі їм довірятимуть.
- `~/.astra/plugin-keys/public.key` — безпечно публікувати.

Додайте `--force`, щоб перезаписати існуючу пару ключів (корисно для ротації — але це анулює вже встановлені довірчі відносини).

### Підпис під час збирання

Окремої команди підпису немає: коли пара ключів існує, `astra-plugin build` підписує автоматично. Архів містить:

- `SIGNATURE` — підписаний маніфест усіх інших файлів у архіві.
- `PUBKEY` — використаний публічний ключ. Користувачі можуть звіряти це зі значенням, якому довіряють (ваш сайт, ваша політика закріплення ключів), перед встановленням.

Щоб опублікувати **непідписану** збірку, видаліть приватний ключ або збирайте на машині, де його немає.

### Перевірка підписів

Перевірка підпису відбувається на боці демона при сайдлоадингу. Демон показує `PUBKEY` бандла в UI Plugins, щоб користувачі могли порівняти відбитки перед натисканням "Install".

## Розповсюдження

### Пряме завантаження

Розповсюджуйте файл `.astraplugin` через свій сайт, GitHub Releases або будь-який файловий хостинг. Користувачі завантажують і перетягують файл на сторінку Plugins в Astra.

### Git + релізні артефакти

Типовий робочий процес релізу:

1. Підніміть `plugin.version` у `plugin.toml`.
2. Закомітьте, поставте тег (`git tag v0.2.0`), запуште.
3. `astra-plugin validate` → `astra-plugin build -o dist/plugin-0.2.0.astraplugin`.
4. Завантажте `.astraplugin` у GitHub Release для цього тегу.

Зручно для CI, оскільки `astra-plugin` — це єдиний бінарник.

### Реєстр

Централізований реєстр плагінів заплановано. Поки його не випустили, діліться плагінами через прямі посилання.

## Сайдлоадинг

Демон надає два RPC для встановлення `.astraplugin`:

- `SideloadPlugin(bytes)` — приймає бандл через gRPC. Використовується файловим вибиральником в UI Astra.
- `ImportPluginFile(path)` — наказує демону прочитати файл з диска. Використовується, коли користувач перетягує файл у UI.

Обидва перевіряють підпис (якщо присутній), валідують маніфест, розпаковують у `~/.astra/plugins/<id>/` та запускають процес.

Видалення плагіна зупиняє процес, видаляє розпакований каталог і очищає стан плагіна.

## Стратегія оновлення

- Підіймайте `plugin.version` при кожному релізі.
- Демон зберігає встановлені версії плагінів і показує значок "Update available", коли новий бандл має вищу SemVer.
- Змінюєте конфігурацію у зворотно несумісний спосіб? Додавайте нові поля зі значеннями за замовчуванням замість перейменування існуючих — демон зберігає стару конфігурацію при оновленнях.

## Локалізація

Додайте каталог `locales/` у свій бандл:

```
locales/
├── en.json
├── ru.json
├── uk.json
├── de.json
├── es.json
├── zh-CN.json
└── ja.json
```

Кожен SDK має хелпер `I18n`, який читає ці файли і м'яко відкатується на невідомих ключах. Маніфест перекладає мітки полів (`ActionType.MyAction`, `FieldLabel.X`) — тримайте ID у коді стабільними, а відображуваний текст — у файлах JSON.

## Чекліст перед релізом

- [ ] `astra-plugin validate` проходить без помилок.
- [ ] `astra-plugin build` виконується успішно й створює архів розумного розміру.
- [ ] У `plugin.toml` є `description`, `author` та `license`.
- [ ] Якщо є `[config]`-схема, кожне поле має розумне значення за замовчуванням.
- [ ] Бандл перевірено сайдлоадингом у чистий екземпляр демона.
- [ ] Відбиток `PUBKEY` задокументовано там, де користувачі можуть його звірити.
- [ ] `locales/` покриває всі рядки, які ваш плагін показує користувачам.
- [ ] `README.md` документує, що робить плагін і які вимоги до рантайму.
- [ ] У вас є спосіб зв'язатися з користувачами, якщо доведеться відкликати скомпрометований реліз.
