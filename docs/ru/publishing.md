# Публикация плагинов

Всё, что нужно, чтобы превратить исходный код в подписанный, распространяемый бандл, который пользователи смогут установить в Astra.

## Бандл `.astraplugin`

`.astraplugin` — это **ZIP-архив** со строго определённой структурой. Демон проверяет манифест, (опционально) проверяет подпись, распаковывает бандл в свой каталог плагинов и запускает `entry.command` с учётными данными.

```
<plugin-id>-<version>.astraplugin
├── plugin.toml                # Манифест
├── bin/                       # Скомпилированный бинарник (только Rust)
│   └── my_plugin.exe
├── dist/                      # Собранный JS (только TypeScript)
│   └── index.js
├── src/                       # Исходники Python (только Python)
│   ├── plugin.py
│   └── __init__.py
├── requirements.txt           # Python-зависимости (только Python)
├── requirements.lock          # Python-зависимости, зафиксированные uv (только Python)
├── ui/                        # Пользовательские UI-файлы (опционально)
├── locales/                   # i18n-файлы JSON (опционально)
├── icon.png | icon.svg        # Опциональный брендинг
├── README.md                  # Опционально
├── LICENSE                    # Опционально
├── SIGNATURE                  # Подпись Ed25519 (если бандл подписан)
└── PUBKEY                     # Публичный ключ Ed25519 (если бандл подписан)
```

Когда `astra-plugin build` создаёт архив, он:

1. Выполняет сборку, специфичную для языка.
2. Копирует собранный артефакт в ожидаемый каталог.
3. Переписывает `entry.command`, чтобы он указывал на путь внутри бандла (только Rust — пути в Python/TS внутри архива стабильны).
4. Добавляет `ui/`, `locales/`, иконку и документацию, если они существуют рядом с `plugin.toml`.
5. Если существует `~/.astra/plugin-keys/private.key`, подписывает каждую запись Ed25519 и добавляет `SIGNATURE` и `PUBKEY` в архив.

## Подписание

### Генерация пары ключей

```bash
astra-plugin keygen
```

Результат:

- `~/.astra/plugin-keys/private.key` — закодированное в base64 Ed25519-начальное значение. **Храните в секрете.** Любой, у кого есть этот файл, сможет подписывать новые версии вашего плагина, и пользователи будут им доверять.
- `~/.astra/plugin-keys/public.key` — можно публиковать.

Добавьте `--force`, чтобы перезаписать существующую пару ключей (полезно при ротации — но это аннулирует уже установленные доверительные отношения).

### Подпись во время сборки

Отдельной команды для подписи нет: если пара ключей существует, `astra-plugin build` подписывает автоматически. Архив содержит:

- `SIGNATURE` — подписанный манифест всех остальных файлов архива.
- `PUBKEY` — использованный публичный ключ. Пользователи могут сверить его с известным корректным значением (ваш сайт, ваша политика закрепления ключей) перед установкой.

Чтобы опубликовать **неподписанную** сборку, либо удалите приватный ключ, либо собирайте на машине, где его нет.

### Проверка подписей

Проверка подписи происходит на стороне демона при сайдлоаде. Демон показывает `PUBKEY` бандла в интерфейсе плагинов, чтобы пользователи могли сравнить отпечатки ключей перед нажатием «Установить».

## Распространение

### Прямая загрузка

Разместите файл `.astraplugin` на вашем сайте, в GitHub Releases или на любом файловом хостинге. Пользователи скачивают файл и перетаскивают его на страницу плагинов Astra.

### Git + артефакты релизов

Типичный рабочий процесс релиза:

1. Увеличьте `plugin.version` в `plugin.toml`.
2. Закоммитьте, поставьте тег (`git tag v0.2.0`), запушьте.
3. `astra-plugin validate` → `astra-plugin build -o dist/plugin-0.2.0.astraplugin`.
4. Загрузите `.astraplugin` в GitHub Release для этого тега.

Удобно для CI, потому что `astra-plugin` — это единый бинарник.

### Реестр

Центральный реестр плагинов запланирован. Пока он не появился, распространяйте плагины через прямые ссылки.

## Сайдлоад

Демон предоставляет два RPC для установки `.astraplugin`:

- `SideloadPlugin(bytes)` — приём бандла через gRPC. Используется файловым пикером в UI Astra.
- `ImportPluginFile(path)` — указывает демону прочитать файл с диска. Используется при перетаскивании файла в UI.

Оба проверяют подпись (если она есть), валидируют манифест, распаковывают в `~/.astra/plugins/<id>/` и запускают процесс.

Удаление плагина останавливает процесс, удаляет распакованный каталог и очищает состояние плагина.

## Стратегия обновлений

- Увеличивайте `plugin.version` для каждого релиза.
- Демон хранит установленные версии плагинов и показывает метку «Доступно обновление», когда новый бандл имеет более высокий SemVer.
- Ломающие изменения конфигурации? Добавляйте новые поля со значениями по умолчанию вместо переименования существующих — демон сохраняет старую конфигурацию при обновлении.

## Локализация

Добавьте каталог `locales/` внутрь вашего бандла:

```
locales/
├── en.json
├── ru.json
├── uk.json
├── de.json
├── es.json
├── zh-CN.json
└── ja.json
```

В каждом SDK есть хелпер `I18n`, который читает эти файлы и корректно обрабатывает неизвестные ключи. Манифест переводит метки полей (`ActionType.MyAction`, `FieldLabel.X`) — оставляйте идентификаторы в коде стабильными, а отображаемый текст держите в JSON-файлах.

## Чек-лист перед релизом

- [ ] `astra-plugin validate` проходит без ошибок.
- [ ] `astra-plugin build` завершается успешно и создаёт архив разумного размера.
- [ ] В `plugin.toml` есть `description`, `author` и `license`.
- [ ] Схема `[config]`, если она есть, имеет разумные значения по умолчанию для каждого поля.
- [ ] Бандл протестирован сайдлоадом в чистый экземпляр демона.
- [ ] Отпечаток `PUBKEY` задокументирован где-то, где пользователи могут его проверить.
- [ ] `locales/` покрывает все строки, которые плагин показывает пользователям.
- [ ] `README.md` документирует, что делает плагин и какие у него требования к среде исполнения.
- [ ] У вас есть способ связаться с пользователями, если потребуется отозвать скомпрометированный релиз.
