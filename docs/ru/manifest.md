# Справочник `plugin.toml`

Каждый плагин описывает себя в манифесте `plugin.toml` в корне своего проекта. Этот файл сообщает демону, как запускать плагин, какие возможности он реализует и какую конфигурацию он принимает.

## Полный пример

```toml
[plugin]
id = "text-utils"
name = "Text Utils"
version = "0.1.1"
description = "Word count, case conversion, regex matching"
author = "Astra Team"
license = "MIT"

[entry]
command = "python"
args = ["-m", "src.plugin"]
runtimes = ["python"]

[capabilities]
tools = true
actions = true
triggers = true
tts = false
stt = false
ai_provider = false
client = false
ui_contributions = false
event_handlers = false

[config]
schema = """
{
  "type": "object",
  "properties": {
    "max_text_length": {
      "type": "number",
      "default": 10000,
      "title": "Max Text Length"
    }
  }
}
"""
```

## `[plugin]` — метаданные

| Поле | Тип | Обязательно | Описание |
| --- | --- | --- | --- |
| `id` | строка | да | Строчные буквы, цифры и дефисы. Уникально идентифицирует плагин. |
| `name` | строка | да | Отображаемое имя в UI плагинов. |
| `version` | строка | да | Семантическая версия (`X.Y.Z`). Используется для определения обновлений. |
| `description` | строка | рекомендуется | Краткое описание. Отображается в UI плагинов. |
| `author` | строка | рекомендуется | Имя автора или организации. |
| `license` | строка | рекомендуется | SPDX-идентификатор лицензии (`MIT`, `Apache-2.0`, …). |

## `[entry]` — как запускать плагин

| Поле | Тип | Обязательно | Описание |
| --- | --- | --- | --- |
| `command` | строка | да | Исполняемый файл для запуска. Для Rust это путь к скомпилированному бинарнику; для Python — `"python"`; для TypeScript — `"node"`. |
| `args` | массив строк | нет | Аргументы, добавляемые перед `--daemon-addr`, `--plugin-id`, `--auth-token`. Типичные значения: `["-m", "src.plugin"]` для Python или `["dist/index.js"]` для Node. |
| `runtimes` | массив строк | нет | Подсказка для демона. Поддерживаются: `"python"`, `"node"`, `"rust"`. Плагины на Python/Node всегда должны указывать это значение, чтобы демон подготовил рантайм (например, создал `uv` venv для Python). |

Демон добавляет к каждому запуску `--daemon-addr <addr> --plugin-id <id>` и опционально `--auth-token <token>` для клиентских плагинов.

## `[capabilities]` — что реализует плагин

Каждое поле — булево, по умолчанию `false`. Устанавливайте `true` только для возможностей, которые ваш код действительно обрабатывает — демон использует это для выделения ресурсов и принятия решения, показывать ли плагин в соответствующем UI (например, добавлять его голоса в выбор TTS).

| Возможность | Назначение |
| --- | --- |
| `tools` | ИИ-инструменты, вызываемые моделью чата. |
| `tts` | Провайдер голосов для преобразования текста в речь. |
| `stt` | Провайдер языков для преобразования речи в текст. |
| `ai_provider` | Альтернативный бэкенд для ИИ-завершений. |
| `actions` | Пользовательские типы действий в Графе команд. |
| `triggers` | Пользовательские типы триггеров в Графе команд. |
| `client` | Полный клиент демона (требует токен сессии). |
| `ui_contributions` | UI-страницы, оверлеи, эффекты, внедрение в слоты. |
| `event_handlers` | Подписка на поток событий демона. |

См. [Возможности](capabilities.md) для полного поведения каждой возможности.

## `[config]` — настройки для пользователя

```toml
[config]
schema = """
{
  "type": "object",
  "properties": {
    "api_key": {
      "type": "string",
      "title": "API Key",
      "description": "Token for the remote service",
      "x-secret": true
    },
    "timeout_ms": {
      "type": "number",
      "default": 5000,
      "minimum": 100,
      "maximum": 60000,
      "title": "Timeout (ms)"
    },
    "mode": {
      "type": "string",
      "enum": ["fast", "accurate"],
      "default": "fast",
      "title": "Mode"
    }
  },
  "required": ["api_key"]
}
"""
```

Правила:

- `schema` — это **строка JSON Schema**. Корень должен иметь `"type": "object"`.
- Демон отрисовывает схему как форму на странице настроек плагина.
- `title` задаёт метку поля; `description` отображается как вспомогательный текст.
- `default` задаёт начальное значение.
- `x-secret: true` маскирует значение в UI и сохраняет его в зашифрованном виде.
- `enum` отображается как выпадающий список.
- Массив `required` помечает обязательные поля.
- Когда пользователь обновляет настройки, демон вызывает `OnConfigChanged` с новым JSON-блобом.

## Поля TOML вне схемы

Плагины могут добавлять произвольные секции верхнего уровня, которые демон игнорирует — это полезно для инструментария. CLI предупреждает о неизвестных полях, но не отвергает их.

## Локализация манифеста

Вы можете поставлять файлы `locales/<lang>.json` внутри бандла плагина, чтобы переводить строки, видимые пользователю (`name`, `description`, метки действий, метки полей и т. д.). Используйте помощник `I18n` в каждом SDK для чтения их из кода. Локализуется только **текст** манифеста — ключи, идентификаторы и значения enum остаются постоянными.

## Валидация манифеста

```bash
astra-plugin validate
```

Проверки:

- Обязательные поля присутствуют (`plugin.id`, `plugin.name`, `plugin.version`, `entry.command`).
- `plugin.id` соответствует шаблону «строчные буквы, цифры и дефисы».
- `plugin.version` соответствует SemVer (`X.Y.Z`).
- Включена хотя бы одна возможность.
- `[config].schema` парсится как JSON и имеет `"type": "object"` в корне.
- Метаданные (description, author) присутствуют.

См. [Справочник CLI → validate](cli.md#astra-plugin-validate).
