# プラグインの公開

ソースコードを、ユーザーが Astra にサイドロードできる署名済みの配布可能なバンドルに変えるために必要なすべてです。

## `.astraplugin` バンドル

`.astraplugin` は特定のレイアウトを持つ **ZIP アーカイブ** です。デーモンはそれを検証し、（任意で）署名を検証し、自分のプラグインディレクトリに展開して、認証情報とともに `entry.command` を起動します。

```
<plugin-id>-<version>.astraplugin
├── plugin.toml                # マニフェスト
├── bin/                       # コンパイル済みバイナリ（Rust のみ）
│   └── my_plugin.exe
├── dist/                      # バンドル済み JS（TypeScript のみ）
│   └── index.js
├── src/                       # Python ソース（Python のみ）
│   ├── plugin.py
│   └── __init__.py
├── requirements.txt           # Python 依存関係（Python のみ）
├── requirements.lock          # uv で解決された Python 依存関係（Python のみ）
├── ui/                        # カスタム UI ファイル（任意）
├── locales/                   # i18n JSON ファイル（任意）
├── icon.png | icon.svg        # 任意のブランディング
├── README.md                  # 任意
├── LICENSE                    # 任意
├── SIGNATURE                  # Ed25519 署名（バンドルが署名済みの場合）
└── PUBKEY                     # Ed25519 公開鍵（バンドルが署名済みの場合）
```

`astra-plugin build` がアーカイブを生成するとき、以下を行います：

1. 言語固有のビルドステップを実行。
2. コンパイル済みの成果物を期待されるディレクトリへコピー。
3. `entry.command` をバンドル内のパスに書き換え（Rust のみ ── Python/TS のアーカイブ内パスは安定しています）。
4. `plugin.toml` の隣に存在すれば `ui/`、`locales/`、アイコン、ドキュメントを追加。
5. `~/.astra/plugin-keys/private.key` があれば、各エントリに Ed25519 で署名し、`SIGNATURE` と `PUBKEY` をアーカイブに追加。

## 署名

### 鍵ペアを生成する

```bash
astra-plugin keygen
```

出力：

- `~/.astra/plugin-keys/private.key` ── base64 エンコードされた Ed25519 シード。**秘密に保管してください。** このファイルを持つ誰でもあなたのプラグインの新バージョンに署名でき、ユーザーはそれを信頼してしまいます。
- `~/.astra/plugin-keys/public.key` ── 公開しても安全です。

既存の鍵ペアを上書きするには `--force` を追加します（ローテーションに便利 ── ただし既に確立した信頼関係を無効化します）。

### ビルド時の署名

別個の署名コマンドはありません。鍵ペアが存在すれば、`astra-plugin build` は自動的に署名します。アーカイブには以下が含まれます：

- `SIGNATURE` ── アーカイブ内の他のすべてのファイルの署名されたマニフェスト。
- `PUBKEY` ── 使用された公開鍵。ユーザーはインストール前に、既知の正しい値（あなたのウェブサイト、鍵ピン留めポリシー）と照合できます。

**署名なし** のビルドを公開するには、秘密鍵を削除するか、秘密鍵のないマシンでビルドします。

### 署名の検証

署名の検証はサイドロード時にデーモン側で行われます。デーモンはプラグイン UI にバンドルの `PUBKEY` を表示するので、ユーザーは「インストール」をクリックする前に指紋を比較できます。

## 配布

### 直接ダウンロード

`.astraplugin` ファイルをあなたのウェブサイト、GitHub Releases、または任意のファイルホストから配布します。ユーザーはダウンロードして Astra のプラグインページにドラッグします。

### Git + リリース成果物

典型的なリリースワークフロー：

1. `plugin.toml` の `plugin.version` を上げる。
2. コミットし、タグ（`git tag v0.2.0`）を付け、プッシュする。
3. `astra-plugin validate` → `astra-plugin build -o dist/plugin-0.2.0.astraplugin`。
4. そのタグの GitHub Release に `.astraplugin` をアップロードする。

`astra-plugin` は単一バイナリなので CI フレンドリーです。

### レジストリ

中央プラグインレジストリは計画中です。出荷されるまでは、直接 URL 経由でプラグインを共有してください。

## サイドロード

デーモンは `.astraplugin` をインストールするための2つの RPC を公開しています：

- `SideloadPlugin(bytes)` ── gRPC 経由でバンドルを受け取ります。Astra UI のファイルピッカーで使用されます。
- `ImportPluginFile(path)` ── デーモンにディスクからファイルを読み込むよう指示します。ユーザーが UI にファイルをドラッグしたときに使用されます。

両方とも（存在すれば）署名を検証し、マニフェストを検証し、`~/.astra/plugins/<id>/` に展開し、プロセスを起動します。

プラグインのアンインストールは、プロセスを停止し、展開ディレクトリを削除し、プラグインの状態をクリアします。

## アップグレード戦略

- リリースごとに `plugin.version` を上げる。
- デーモンはインストール済みのプラグインバージョンを保存し、新しいバンドルがより高い SemVer を持つときに「アップデートあり」バッジを表示します。
- 破壊的な設定変更がありますか？ 既存フィールドをリネームするのではなく、デフォルト付きの新しいフィールドを追加してください ── デーモンはアップグレード時に古い設定を保持します。

## ローカライズ

バンドル内に `locales/` ディレクトリを同梱します：

```
locales/
├── en.json
├── ru.json
├── uk.json
├── de.json
├── es.json
├── zh-CN.json
└── ja.json
```

各 SDK には、これらのファイルを読み込み、未知のキーで安全にフォールバックする `I18n` ヘルパーがあります。マニフェストはフィールドラベル（`ActionType.MyAction`、`FieldLabel.X`）を翻訳します ── コード中の ID は安定に保ち、表示テキストは JSON ファイルに入れてください。

## リリース前チェックリスト

- [ ] `astra-plugin validate` がエラーなく通る。
- [ ] `astra-plugin build` が成功し、妥当なサイズのアーカイブを生成する。
- [ ] `plugin.toml` に `description`、`author`、`license` がある。
- [ ] `[config]` schema があれば、各フィールドに妥当なデフォルトがある。
- [ ] クリーンなデーモンインスタンスにサイドロードしてバンドルをテスト済み。
- [ ] `PUBKEY` の指紋がユーザーが検証できる場所にドキュメント化されている。
- [ ] `locales/` がプラグインがユーザーに表示するすべての文字列をカバーしている。
- [ ] `README.md` にプラグインが何をするか、ランタイム要件があるかが書かれている。
- [ ] 侵害されたリリースを取り下げる必要が生じたとき、ユーザーに連絡する方法がある。
